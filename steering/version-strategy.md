## Output Format Guidelines

### Shell Script Output Principles

When implementing the CDK Upgrader workflows, follow these output formatting principles:

**DO**:
- Use simple `echo` statements for status updates
- Keep output concise and readable
- Use consistent formatting (e.g., `=== Section Name ===`)
- Return detailed results in the agent's response, not in shell output
- **ALWAYS include proper error handling with `set -euo pipefail`**
- **ALWAYS validate variables before arithmetic operations**
- **ALWAYS use cross-platform compatible date functions**

**DON'T**:
- Create elaborate ASCII art or box drawings in shell scripts
- Generate multi-line formatted tables in shell output
- Use complex formatting that makes scripts hard to read
- Output verbose summaries that should be in the agent response
- **NEVER ignore shell script errors or use hardcoded fallbacks**
- **NEVER use platform-specific date commands without compatibility checks**

**Example - Good**:
```bash
#!/bin/bash
set -euo pipefail  # REQUIRED: Strict error handling

echo "=== Target Version Selection ==="
echo "TARGET_CDK_CLI_VERSION: $CLI_VERSION"
echo "TARGET_CDK_LIB_VERSION: $LIB_VERSION"
echo "Selection complete."
```

**Example - Avoid**:
```bash
#!/bin/bash
# Missing error handling - WRONG!

echo "╔══════════════════════════════════════════════════════════════════╗"
echo "║                    TARGET VERSION SUMMARY                        ║"
echo "╠══════════════════════════════════════════════════════════════════╣"
# ... many more lines of complex formatting

# Hardcoded fallback without proper error handling - WRONG!
TARGET_CDK_CLI_VERSION="2.1034.0"  # This should never be hardcoded
```

### Agent Response Format

Detailed analysis, summaries, and formatted reports should be provided in the agent's response text, not generated by shell scripts. This keeps scripts focused on execution while providing rich feedback to users.

## CDK Version Independence

### Overview

Since February 2025, AWS CDK has adopted an independent versioning model where the CDK CLI and CDK Construct Library have separate version numbers. Understanding this model is essential for successful CDK upgrades.

### CDK CLI Versions (2.1000.x+ Series)

The CDK CLI (Command Line Interface) is the tool used to synthesize, deploy, and manage CDK applications.

**Version Series**: `2.1000.x`, `2.1001.x`, `2.1002.x`, ...

**Key characteristics**:
- Versions start from the 2.1000.x series (post-February 2025)
- Released independently from the Construct Library
- Backward compatible with older Construct Library versions
- Always safe to use the latest CLI version

**Package name**: `aws-cdk` (in devDependencies)

**Example**:
```json
{
  "devDependencies": {
    "aws-cdk": "^2.1005.0"
  }
}
```

### CDK Construct Library Versions (2.175.x+ Series)

The CDK Construct Library (`aws-cdk-lib`) contains all the AWS service constructs used to define infrastructure.

**Version Series**: `2.175.x`, `2.176.x`, `2.177.x`, ...

**Key characteristics**:
- Continues the original CDK v2 versioning scheme
- Contains AWS service constructs and L1/L2/L3 patterns
- May introduce breaking changes in minor versions
- New features and constructs added in each release

**Package name**: `aws-cdk-lib` (in dependencies)

**Example**:
```json
{
  "dependencies": {
    "aws-cdk-lib": "^2.175.3"
  }
}
```

### Compatibility Rule

**Critical Rule**: CLI release date must be greater than or equal to Construct Library release date.

```
CLI Release Date >= Construct Library Release Date
```

**Why this matters**:
- The CLI must understand all constructs in the library
- Newer CLI versions include synthesis improvements for newer constructs
- Using an older CLI with a newer library may cause synthesis failures

**Compatibility examples**:

| CLI Version | CLI Release | Lib Version | Lib Release | Compatible? |
|-------------|-------------|-------------|-------------|-------------|
| 2.1005.0    | 2025-03-15  | 2.175.3     | 2025-03-10  | ✓ Yes       |
| 2.1005.0    | 2025-03-15  | 2.176.0     | 2025-03-20  | ✗ No        |
| 2.1010.0    | 2025-04-01  | 2.176.0     | 2025-03-20  | ✓ Yes       |

### Version Display Format

When displaying version information, always show both versions separately:

```
=== CDK Version Information ===

CDK CLI Version:       2.1005.0 (latest)
CDK Lib Version:       2.175.3 (previous stable)
Latest Lib Available:  2.176.0

Compatibility: ✓ CLI release date (2025-03-15) >= Lib release date (2025-03-10)
```

### Pre-February 2025 Projects

Projects created before February 2025 will have synchronized version numbers:

```json
{
  "dependencies": {
    "aws-cdk-lib": "^2.150.0"
  },
  "devDependencies": {
    "aws-cdk": "^2.150.0"
  }
}
```

After upgrade, these will have independent versions:

```json
{
  "dependencies": {
    "aws-cdk-lib": "^2.175.3"
  },
  "devDependencies": {
    "aws-cdk": "^2.1005.0"
  }
}
```

## Version Selection Algorithm

### Overview

The CDK Upgrader uses a **5-step version selection algorithm** to balance stability with access to recent features and bug fixes. The algorithm follows a strict order: **first determine CLI version, then use CLI release date as the constraint for selecting Lib version**.

**CRITICAL: 5-Step Algorithm Validation**

**Before executing version selection, the agent MUST announce**:

```bash
echo "=== CDK Upgrader Power: Version Selection ==="
echo "Following: version-strategy.md 5-step algorithm"
echo "Step 1: Select CLI version (use latest stable if > 7 days old, otherwise previous stable)"  
echo "Step 2: Record CLI release date as compatibility constraint"
echo "Step 3: Select Lib version (use latest stable if > 7 days old, otherwise previous stable)"
echo "Step 4: MANDATORY compatibility check (CLI release date >= Lib release date)"
echo "Step 5: Final validation of both versions"
echo ""
echo "⚠️  WARNING: Skipping Step 4 will result in incompatible versions!"
echo "⚠️  All 5 steps MUST be executed in sequence!"
```

**RED FLAGS - STOP EXECUTION IF YOU SEE**:
- Direct usage of `npm view aws-cdk version` without the full 5-step algorithm
- Direct usage of `npm view aws-cdk-lib version` without the full 5-step algorithm  
- **Compatibility check (Step 4) being skipped or omitted**
- Missing cross-platform date calculation functions
- Hardcoded version fallbacks without proper validation

**CRITICAL IMPLEMENTATION REQUIREMENT**: If compatibility check (Step 4) is skipped, the selected versions will be incompatible and cause synthesis failures.

### Strategy Summary

| Step | Package       | Selection Strategy                                                    |
|------|---------------|-----------------------------------------------------------------------|
| 1-2  | `aws-cdk`     | Use latest stable version if > 7 days old, otherwise previous stable version. Record CLI release date. |
| 3    | `aws-cdk-lib` | Use latest stable version if > 7 days old, otherwise previous stable version |
| 4    | Compatibility | MANDATORY: Verify CLI release date >= Lib release date |
| 5    | Validation    | Final verification of both versions |

### Core Selection Rules

**5-Step Algorithm:**
1. **Step 1**: Select CLI version - use latest stable if > 7 days old, otherwise previous stable
2. **Step 2**: Record CLI release date as compatibility constraint
3. **Step 3**: Select Lib version - use latest stable if > 7 days old, otherwise previous stable
4. **Step 4**: MANDATORY compatibility check - CLI release date >= Lib release date
5. **Step 5**: Final validation of both versions

**Additional Constraints:**
- **Always select highest patch within a minor series** - Get all bug fixes
- **Template Generation is Mandatory** - Always generate CDK template for comparison

### Target Version Variables

The algorithm produces two target versions that are used throughout the upgrade process:

- **TARGET_CDK_CLI_VERSION**: The CDK CLI version to use (e.g., `2.1005.0`)
- **CLI_RELEASE_DATE**: The release date of TARGET_CDK_CLI_VERSION (used as constraint)
- **TARGET_CDK_LIB_VERSION**: The CDK Construct Library version to use (e.g., `2.175.3`)

---

### Step 1: CDK CLI Version Selection (TARGET_CDK_CLI_VERSION)

**Rule**: Use latest stable version if > 7 days old. If latest version is < 7 days old, find the previous stable version that is > 7 days old.

**Rationale**:
- Use the most recent stable version available
- Highest patch includes all bug fixes for that minor version
- 7-day buffer provides stability assurance
- If latest patch in a minor series is unstable, the entire series is considered unstable

**Algorithm**:
```bash
# Cross-platform date calculation function
calculate_days_since() {
    local release_date="$1"
    local current_timestamp=$(date +%s)
    
    # Handle different date formats and OS compatibility
    if [[ "$OSTYPE" == "darwin"* ]]; then
        # macOS
        local release_timestamp=$(date -j -f "%Y-%m-%dT%H:%M:%S" "${release_date%.*}" +%s || \
                                 date -j -f "%Y-%m-%d" "${release_date%%T*}" +%s || \
                                 echo "0")
    else
        # Linux
        local release_timestamp=$(date -d "$release_date" +%s || echo "0")
    fi
    
    if [ "$release_timestamp" -eq 0 ]; then
        echo "ERROR: Unable to parse date: $release_date" >&2
        return 1
    fi
    
    echo $(( (current_timestamp - release_timestamp) / 86400 ))
}

# Main algorithm steps:
1. Query npm registry for all aws-cdk versions and release dates
2. Get the latest version (highest version number)
3. Check if latest version is > 7 days old using cross-platform date calculation
4. If yes: TARGET_CDK_CLI_VERSION = latest version
5. If no: Find previous stable version > 7 days old
6. Record CLI_RELEASE_DATE = release date of TARGET_CDK_CLI_VERSION
```

**Example (latest version is stable)**:
```bash
# Cross-platform compatible version selection
ALL_CLI_VERSIONS=$(npm view aws-cdk versions --json)
LATEST_CLI_VERSION=$(npm view aws-cdk version)

# Get release date with error handling
LATEST_CLI_DATE=$(npm view aws-cdk time --json | jq -r ".[\"$LATEST_CLI_VERSION\"]")
if [ -z "$LATEST_CLI_DATE" ] || [ "$LATEST_CLI_DATE" = "null" ]; then
    echo "ERROR: Unable to get release date for $LATEST_CLI_VERSION" >&2
    exit 1
fi

# Calculate days since release (cross-platform)
DAYS_SINCE_CLI=$(calculate_days_since "$LATEST_CLI_DATE")
if [ $? -ne 0 ]; then
    echo "ERROR: Date calculation failed" >&2
    exit 1
fi

echo "Latest CLI version: $LATEST_CLI_VERSION"
echo "Release date: $LATEST_CLI_DATE"
echo "Days since release: $DAYS_SINCE_CLI"

if [ "$DAYS_SINCE_CLI" -gt 7 ]; then
    echo "✓ Latest version is more than 7 days old"
    TARGET_CDK_CLI_VERSION="$LATEST_CLI_VERSION"
else
    echo "⚠ Latest version is less than 7 days old, finding previous stable version..."
    
    # Find previous stable version (implement proper algorithm instead of hardcoding)
    PREVIOUS_STABLE=$(find_previous_stable_version "$ALL_CLI_VERSIONS" "$LATEST_CLI_VERSION")
    if [ -z "$PREVIOUS_STABLE" ]; then
        echo "ERROR: No stable version found" >&2
        exit 1
    fi
    
    TARGET_CDK_CLI_VERSION="$PREVIOUS_STABLE"
fi

CLI_RELEASE_DATE=$(npm view aws-cdk time --json | jq -r ".[\"$TARGET_CDK_CLI_VERSION\"]")
echo "Selected TARGET_CDK_CLI_VERSION: $TARGET_CDK_CLI_VERSION"
echo "CLI_RELEASE_DATE: $CLI_RELEASE_DATE"
```

### Step 2: CDK Construct Library Version Selection (TARGET_CDK_LIB_VERSION)

**Rule**: Use latest version if > 7 days old and compatible with CLI. If latest version is < 7 days old, find the previous stable version that is > 7 days old and compatible.

**Rationale**:
- Use the most recent stable version available
- Highest patch includes all bug fixes for that minor version
- 7-day buffer provides stability assurance
- If latest patch in a minor series is unstable, the entire series is considered unstable
- Must be compatible with selected CLI version

**Algorithm**:
```bash
# Cross-platform date calculation (reuse function from CLI selection)
calculate_days_since() {
    local release_date="$1"
    local current_timestamp=$(date +%s)
    
    if [[ "$OSTYPE" == "darwin"* ]]; then
        # macOS
        local release_timestamp=$(date -j -f "%Y-%m-%dT%H:%M:%S" "${release_date%.*}" +%s || \
                                 date -j -f "%Y-%m-%d" "${release_date%%T*}" +%s || \
                                 echo "0")
    else
        # Linux
        local release_timestamp=$(date -d "$release_date" +%s || echo "0")
    fi
    
    if [ "$release_timestamp" -eq 0 ]; then
        echo "ERROR: Unable to parse date: $release_date" >&2
        return 1
    fi
    
    echo $(( (current_timestamp - release_timestamp) / 86400 ))
}

# Main algorithm steps:
1. Query npm registry for all aws-cdk-lib versions and release dates
2. Get the latest version (highest version number)
3. Check if latest version is > 7 days old using cross-platform date calculation
4. If yes: CANDIDATE_LIB_VERSION = latest version
5. If no: Find previous stable version > 7 days old
6. Verify final compatibility: CANDIDATE_LIB_VERSION release date <= CLI_RELEASE_DATE
7. If yes: TARGET_CDK_LIB_VERSION = CANDIDATE_LIB_VERSION
8. If no: Continue to previous version and repeat from step 3
```

**Example (latest version is stable and compatible)**:
```bash
# Get all lib versions and latest
ALL_LIB_VERSIONS=$(npm view aws-cdk-lib versions --json)
LATEST_LIB_VERSION=$(npm view aws-cdk-lib version)

echo "All Lib versions: $(echo "$ALL_LIB_VERSIONS" | jq -r '.[-10:]' | tr '\n' ' ')"
echo "Latest version: $LATEST_LIB_VERSION"

# Get release date with error handling
LATEST_LIB_DATE=$(npm view aws-cdk-lib time --json | jq -r ".[\"$LATEST_LIB_VERSION\"]")
if [ -z "$LATEST_LIB_DATE" ] || [ "$LATEST_LIB_DATE" = "null" ]; then
    echo "ERROR: Unable to get release date for $LATEST_LIB_VERSION" >&2
    exit 1
fi

# Calculate days since release
DAYS_SINCE_LIB=$(calculate_days_since "$LATEST_LIB_DATE")
if [ $? -ne 0 ]; then
    echo "ERROR: Date calculation failed for lib version" >&2
    exit 1
fi

echo "Check $LATEST_LIB_VERSION release date ($LATEST_LIB_DATE):"
echo "Days since release: $DAYS_SINCE_LIB"

if [ "$DAYS_SINCE_LIB" -gt 7 ]; then
    echo "✓ Latest lib version is more than 7 days old"
    CANDIDATE_LIB_VERSION="$LATEST_LIB_VERSION"
else
    echo "⚠ Latest lib version is less than 7 days old, finding previous stable version..."
    CANDIDATE_LIB_VERSION=$(find_previous_stable_lib_version "$ALL_LIB_VERSIONS" "$LATEST_LIB_VERSION")
fi

# Final compatibility check with CLI_RELEASE_DATE
CANDIDATE_LIB_DATE=$(npm view aws-cdk-lib time --json | jq -r ".[\"$CANDIDATE_LIB_VERSION\"]")
CLI_TIMESTAMP=$(date -d "$CLI_RELEASE_DATE" +%s || date -j -f "%Y-%m-%dT%H:%M:%S" "${CLI_RELEASE_DATE%.*}" +%s)
LIB_TIMESTAMP=$(date -d "$CANDIDATE_LIB_DATE" +%s || date -j -f "%Y-%m-%dT%H:%M:%S" "${CANDIDATE_LIB_DATE%.*}" +%s)

if [ "$LIB_TIMESTAMP" -le "$CLI_TIMESTAMP" ]; then
    echo "✓ Compatibility check passed"
    TARGET_CDK_LIB_VERSION="$CANDIDATE_LIB_VERSION"
else
    echo "✗ Compatibility check failed, finding earlier version..."
    # Continue searching backwards
fi

echo "TARGET_CDK_LIB_VERSION: $TARGET_CDK_LIB_VERSION"
```

**Example (latest minor too new, skip entire series)**:
```
Minor series check (version number descending):

2.233.x → highest patch: 2.233.0 (2024-12-30, 2 days ago)
  - Stability: 2 days ≤ 7 days ✗ → SKIP entire 2.233.x series

2.232.x → highest patch: 2.232.2 (2024-12-20, 12 days ago)
  - Stability: 12 days > 7 days ✓
  - Compatibility: 2024-12-20 <= CLI_RELEASE_DATE ✓
  → FOUND!

TARGET_CDK_LIB_VERSION: 2.232.2
```

**CRITICAL**: When searching for compatible versions:
- Check each **minor series** in descending order (2.233.x, 2.232.x, 2.231.x...)
- For each minor, check only the **highest patch** version
- If highest patch fails → skip entire minor series
- Select the first minor series where highest patch passes both rules

### Step 3: Compatibility Validation (CRITICAL - NEVER SKIP)

**Rule**: Verify compatibility constraint is satisfied.

**⚠️ CRITICAL**: This step MUST NOT be skipped. If skipped, incompatible versions will be selected.

**Algorithm**:
```
1. MANDATORY: Verify TARGET_CDK_LIB_VERSION release date <= CLI_RELEASE_DATE
2. If compatibility check passes: versions are final
3. If compatibility check fails: MUST continue searching backwards until compatible version found
4. If no compatible version found: MUST report error and abort
```

**Finding Compatible Version Algorithm**:
```bash
# When latest Lib version is incompatible, find compatible version
# Strategy: Check highest patch of each minor series, version number descending
find_compatible_lib_version() {
    local cli_release_date="$1"
    local cli_timestamp=$(date -d "$cli_release_date" +%s || \
        date -j -f "%Y-%m-%dT%H:%M:%S" "${cli_release_date%.*}" +%s)
    
    # Get all versions
    local all_versions=$(npm view aws-cdk-lib versions --json)
    
    # Get unique minor versions, sorted descending (e.g., 2.233, 2.232, 2.231...)
    local minor_versions=$(echo "$all_versions" | jq -r '.[]' | \
        grep -E '^2\.[0-9]+\.[0-9]+$' | \
        cut -d. -f1,2 | sort -t. -k2 -nr | uniq)
    
    for minor in $minor_versions; do
        # Find highest patch in this minor series
        local highest_patch=$(echo "$all_versions" | jq -r '.[]' | \
            grep "^${minor}\." | sort -t. -k3 -nr | head -1)
        
        [ -z "$highest_patch" ] && continue
        
        # Get release date for highest patch
        local patch_date=$(npm view aws-cdk-lib time."$highest_patch")
        local patch_timestamp=$(date -d "$patch_date" +%s || \
            date -j -f "%Y-%m-%dT%H:%M:%S" "${patch_date%.*}" +%s)
        
        # Check compatibility: patch release date <= CLI release date
        if [ "$patch_timestamp" -le "$cli_timestamp" ]; then
            # Check 7-day stability rule
            local days_old=$(calculate_days_since "$patch_date")
            if [ "$days_old" -gt 7 ]; then
                echo "$highest_patch"
                return 0
            fi
            # If highest patch is too new (< 7 days), skip entire minor series
            # because lower patches are older and less desirable
        fi
        
        # This minor series doesn't have a compatible stable version
        # Try previous minor series
    done
    
    echo "ERROR: No compatible stable version found" >&2
    return 1
}
```

**CRITICAL**: Version selection strategy:
1. Group versions by **minor version** (e.g., 2.232.x, 2.231.x)
2. For each minor series (descending order), check the **highest patch**
3. If highest patch is compatible AND stable → SELECT IT
4. If highest patch is incompatible → SKIP entire minor series (try previous minor)
5. If highest patch is compatible but too new (< 7 days) → SKIP entire minor series

**Why version number order (not release date)**:
- Within a minor series, patches are typically released in order
- If 2.232.2 is incompatible, 2.232.1 and 2.232.0 are likely also incompatible or older
- Checking by minor series is more efficient and logical
- Users want the highest patch (most bug fixes) in a compatible minor series

**Example of FAILED compatibility check**:
```bash
CLI_DATE="2025-12-17T14:36:35.919Z"

# Check minor series in descending order:
# 2.233.x → highest patch: 2.233.0 (2025-12-18) > CLI date ✗ → SKIP entire 2.233.x
# 2.232.x → highest patch: 2.232.2 (2025-12-12) <= CLI date ✓, 22 days old > 7 ✓ → FOUND!

TARGET_CDK_LIB_VERSION=2.232.2
```

**Example when entire minor series is incompatible**:
```bash
CLI_DATE="2025-12-05T00:00:00.000Z"

# Check minor series in descending order:
# 2.233.x → highest patch: 2.233.0 (2025-12-18) > CLI date ✗ → SKIP
# 2.232.x → highest patch: 2.232.2 (2025-12-12) > CLI date ✗ → SKIP entire 2.232.x
# 2.231.x → highest patch: 2.231.1 (2025-12-01) <= CLI date ✓, 35 days old > 7 ✓ → FOUND!

TARGET_CDK_LIB_VERSION=2.231.1
```

**Scenarios requiring fallback**:

### Cross-Platform Date Function

**REQUIRED**: All version selection scripts MUST use this cross-platform date function:

```bash
# Cross-platform date calculation function
calculate_days_since() {
    local release_date="$1"
    
    # Get current timestamp with validation
    local current_timestamp=$(date +%s)
    local current_year=$(date +%Y)
    
    # Validate current year is reasonable (2024-2030 range)
    if [ "$current_year" -lt 2024 ] || [ "$current_year" -gt 2030 ]; then
        echo "ERROR: System date appears incorrect. Current year: $current_year" >&2
        echo "Please check system date and time settings" >&2
        return 1
    fi
    
    # Handle different date formats and OS compatibility
    if [[ "$OSTYPE" == "darwin"* ]]; then
        # macOS - try multiple date formats
        local release_timestamp
        
        # Try ISO format with milliseconds (2025-12-17T14:36:35.919Z)
        release_timestamp=$(date -j -f "%Y-%m-%dT%H:%M:%S" "${release_date%.*}" +%s)
        
        # Try date only format (2025-12-17)
        if [ -z "$release_timestamp" ] || [ "$release_timestamp" = "0" ]; then
            release_timestamp=$(date -j -f "%Y-%m-%d" "${release_date%%T*}" +%s)
        fi
        
        # Fallback: try without format specifier
        if [ -z "$release_timestamp" ] || [ "$release_timestamp" = "0" ]; then
            release_timestamp=$(date -j -f "%Y-%m-%d" "${release_date}" +%s || echo "0")
        fi
    else
        # Linux - GNU date handles most formats automatically
        local release_timestamp=$(date -d "$release_date" +%s || echo "0")
    fi
    
    # Validate timestamp
    if [ -z "$release_timestamp" ] || [ "$release_timestamp" = "0" ]; then
        echo "ERROR: Unable to parse date: $release_date" >&2
        echo "OS: $OSTYPE" >&2
        return 1
    fi
    
    # Calculate days difference
    local days_diff=$(( (current_timestamp - release_timestamp) / 86400 ))
    
    # Debug output for troubleshooting
    echo "DEBUG: Current timestamp: $current_timestamp ($(date))" >&2
    echo "DEBUG: Release timestamp: $release_timestamp ($release_date)" >&2
    echo "DEBUG: Days difference: $days_diff" >&2
    
    echo "$days_diff"
}

# Usage example with error handling
check_version_age() {

    fi
```

#### Parsing Semantic Versions

CDK uses semantic versioning: `MAJOR.MINOR.PATCH`

**Parsing algorithm**:
```
Input: "2.175.3"

1. Split by "." → ["2", "175", "3"]
2. Parse as integers:
   - major = 2
   - minor = 175
   - patch = 3
```

**Handling pre-release versions**:
```
Input: "2.176.0-alpha.0"

1. Split by "-" → ["2.176.0", "alpha.0"]
2. Take first part: "2.176.0"
3. Mark as pre-release (exclude from selection)
```

#### Version Comparison

To compare versions, compare components in order: major → minor → patch

**Comparison rules**:
```
2.176.0 > 2.175.3  (minor 176 > 175)
2.175.3 > 2.175.2  (patch 3 > 2)
2.175.3 > 2.174.5  (minor 175 > 174)
```

**Comparison algorithm**:
```
function compareVersions(v1, v2):
  if v1.major != v2.major:
    return v1.major - v2.major
  if v1.minor != v2.minor:
    return v1.minor - v2.minor
  return v1.patch - v2.patch
```
